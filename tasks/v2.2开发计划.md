# Godot AI Assistant v2.2 开发计划

## 1. 版本目标

v2.2 的目标不是继续横向堆功能，而是把 v2.1 已经成型的架构打磨成“更稳、更可解释、更可维护”的运行时。

这个版本聚焦五件事：

1. Runtime 状态机化
2. Apply 可回滚
3. 上下文相关性排序第一版
4. 网络层重试与降级
5. 调试与可观测性增强

这五项完成后，插件会从“结构已经分层”进入“运行时开始产品化”的阶段。

---

## 2. 版本边界

v2.2 明确不做这些事情：

- 不引入完整工具调用协议
- 不引入多 Agent 或后台自治任务
- 不扩展成通用系统级 Agent
- 不大幅改 UI 外观
- 不追求一次性支持很多新 provider 特性

换句话说，v2.2 重点是打底，不是扩张。

---

## 3. 版本完成标准

如果 v2.2 完成，应该达到这些效果：

1. 一次请求从准备到结束的状态更清楚，UI 和调试面板都能解释正在发生什么
2. AI 代码应用后可以回滚，不再只有“应用成功/失败”两个结果
3. 上下文选择不再只是固定优先级，而开始和用户当前请求相关
4. 流式请求异常时，体验更稳，不会轻易整段失败
5. 维护者能从调试信息里看懂“为什么它选了这些上下文、为什么它给了这个动作计划”

---

## 4. 任务拆分总览

建议按下面顺序推进：

1. Runtime 状态机化
2. 调试信息增强
3. 网络层重试与降级
4. Apply 可回滚
5. 上下文相关性排序
6. 回归清单更新与验证

这个顺序的原因是：

- Runtime 状态和调试信息是后续所有改动的基础
- 网络和 Apply 是最容易让用户直接感知质量差异的部分
- 相关性排序适合在可观测性增强之后做，否则难以判断效果

---

## 5. 任务一：Runtime 状态机化

## 5.1 目标

把 `AIRuntime` 从“带状态字段的编排器”升级成“有清晰阶段边界的运行时状态机”。

## 5.2 涉及文件

- [ai_runtime.gd](/E:/test/addons/ai_assistant/core/ai_runtime.gd)
- [ai_dock.gd](/E:/test/addons/ai_assistant/ai_dock.gd)

## 5.3 要做的改动

- 明确定义请求生命周期阶段常量
- 为每个阶段提供统一的切换入口
- 给每次请求生成 `request_id`
- 在 `last_request_preview` 或新的运行时快照中记录阶段变化
- 把“正在准备”“正在构造上下文”“正在请求模型”“正在规划动作”“等待确认”“执行中”区分开
- 避免 UI 自己猜当前是不是某种中间状态

## 5.4 建议状态

- `idle`
- `collect_context`
- `load_rules`
- `build_request`
- `send_request`
- `streaming`
- `normalize_response`
- `plan_action`
- `await_review`
- `applying`
- `persisting`
- `completed`
- `failed`
- `stopped`

## 5.5 验收标准

- 发送一次请求时，状态会按预期变化
- 停止流式请求时，状态最终回到 `stopped` 或 `idle`
- 动作预览/确认阶段不会再混同于普通“忙碌中”
- Dock 能只依赖 Runtime 暴露的状态而更新按钮和提示

---

## 6. 任务二：调试与可观测性增强

## 6.1 目标

让维护者能从 UI 里看懂 Runtime 正在做什么，而不是只能靠读代码猜。

## 6.2 涉及文件

- [ai_runtime.gd](/E:/test/addons/ai_assistant/core/ai_runtime.gd)
- [ai_dock.gd](/E:/test/addons/ai_assistant/ai_dock.gd)
- [ai_prompt_builder.gd](/E:/test/addons/ai_assistant/prompt/ai_prompt_builder.gd)
- [ai_context_builder.gd](/E:/test/addons/ai_assistant/prompt/ai_context_builder.gd)

## 6.3 要做的改动

- 在调试视图中展示 `request_id`
- 展示当前 Runtime 状态
- 展示当前模型 profile 与 capability
- 展示最终选中的上下文项
- 展示被丢弃的上下文项及原因
- 展示动作规划摘要
- 展示最近失败原因
- 展示自动压缩是否发生

## 6.4 建议输出内容

- 请求基础信息
- provider / model / url 类型
- context budget 总量与剩余量
- selected context manifest
- dropped context manifest
- action plan summary
- last error

## 6.5 验收标准

- 一次正常请求能看到所选上下文项
- 一次超预算请求能看到哪些上下文被裁剪
- 一次失败请求能看到清楚的失败原因，而不只是通用提示
- 一次可 Apply 的回复能看到动作计划概要

## 7. 任务三：网络层重试与降级

## 7.1 目标

提高模型请求的稳定性，减少因为流式网络抖动导致的整次失败。

## 7.2 涉及文件

- [ai_net_client.gd](/E:/test/addons/ai_assistant/ai_net_client.gd)
- [ai_runtime.gd](/E:/test/addons/ai_assistant/core/ai_runtime.gd)
- [ai_provider_adapter.gd](/E:/test/addons/ai_assistant/net/ai_provider_adapter.gd)
- [ai_provider_profiles.gd](/E:/test/addons/ai_assistant/net/ai_provider_profiles.gd)

## 7.3 要做的改动

- 支持单次自动重试
- 对连接失败、首包超时、流中断分别处理
- 当 provider 支持时，允许从 streaming 降级到 non-streaming
- 支持把部分已收到的内容保留为“部分响应”
- 在调试信息里记录是否发生 retry 或 fallback

## 7.4 建议策略

- 连接阶段失败：允许重试一次
- 请求已发出但长期无首包：允许重试一次
- 流中断且已收到部分内容：优先保留部分内容并提示
- 流中断且 provider 可降级：尝试降级为非流式
- HTTP 4xx：不自动重试
- HTTP 5xx：可考虑单次重试

## 7.5 验收标准

- 短暂网络问题不会立刻把请求彻底打死
- provider 不支持流式降级时也不会错误走 fallback
- 出现 partial content 时，用户看到的是明确说明，不是静默吞掉
- 调试面板能看到 retry/fallback 是否发生

## 8. 任务四：Apply 可回滚

## 8.1 目标

让 AI 修改具备最小闭环：

- 计划
- 预览
- 确认
- 应用
- 记录
- 回滚

## 8.2 涉及文件

- [ai_action_executor.gd](/E:/test/addons/ai_assistant/actions/ai_action_executor.gd)
- [ai_dock.gd](/E:/test/addons/ai_assistant/ai_dock.gd)
- [ai_storage.gd](/E:/test/addons/ai_assistant/ai_storage.gd)
- [ai_runtime.gd](/E:/test/addons/ai_assistant/core/ai_runtime.gd)

## 8.3 要做的改动

- 为每次已执行的 AI 变更记录一份变更记录
- 记录目标文件或资源路径
- 记录动作类型
- 记录变更前文本
- 记录变更后文本
- 记录风险等级和时间戳
- 提供“撤销上次 AI 改动”的 UI 入口

## 8.4 最小实现建议

v2.2 不需要做完整历史版本管理，只做：

- 当前会话最近一次 AI 修改可回滚
- 仅支持文本型脚本和场景文本型变更回滚
- 回滚前同样提供确认提示

## 8.5 验收标准

- 一次成功 Apply 后会生成回滚记录
- 回滚操作能恢复目标文本
- 若目标已被用户再次修改，至少给出冲突提示而不是静默覆盖
- 高风险整文件替换同样能留下回滚信息

## 9.2 涉及文件

- [ai_context_builder.gd](/E:/test/addons/ai_assistant/prompt/ai_context_builder.gd)

## 9.3 要做的改动

- 当用户明确提到路径、类名、函数名时，提高对应文件或上下文优先级
- 提升最近编辑文件、最近失败文件的权重

- 用户提到某个文件时，该文件上下文优先进入请求
- 用户聚焦当前选区时，不会被项目地图挤掉
- 用户问 Git 相关问题时，Git 摘要更容易入选
- 被裁剪项和入选项在调试面板中可解释

## 13. 风险提示

v2.2 里最容易踩坑的点有这些：

### 13.1 状态机改造过度耦合 UI

如果 Runtime 状态定义不稳定，Dock 会跟着频繁改。  
解决方式：先把 Runtime 状态和输出 schema 固定下来，再接 UI。

### 13.2 回滚记录不完整

如果 Apply 后没有保存足够上下文，撤销很容易变成“理论支持，实际不可用”。  
解决方式：先做文本级最小闭环，不要一开始追求复杂版本管理。

### 13.3 相关性排序误伤现有上下文

如果动态评分过强，可能把稳定有效的基础上下文挤掉。  
解决方式：保留最低保障预算，不让高优先级上下文完全消失。

### 13.4 重试与 fallback 导致重复内容

如果流式失败后直接重发，可能生成重复文本。  
解决方式：在 Runtime 层对 partial content 和 fallback 行为做明确分流。

## 14. 里程碑定义

### M1：Runtime 可观测

标志：

- Runtime 状态机完成
- 调试信息可看到完整请求阶段

### M2：请求更稳

标志：

- retry/fallback 生效
- 失败原因更清晰

### M3：Apply 闭环

标志：

- AI 修改可回滚
- 风险与变更记录更完整

### M4：上下文更准

标志：

- 动态相关性排序投入使用
- 调试面板能解释上下文入选原因

### M5：可发布

标志：

- checklist 更新完成
- headless 验证通过
- README / 任务文档更新完成

## 15. 本版本完成后的预期结果

v2.2 做完之后，项目应该具备下面这些更成熟的特征：

- 运行时更像一个稳定内核，而不是一段请求流程
- AI 修改更像“有安全闭环的编辑器操作”，而不是一次性写入
- 上下文更开始“理解当前请求”，而不是机械拼接
- 网络波动和流式异常时，用户体验更稳
- 维护者更容易看懂系统行为和排查问题

这会为 v2.3 的 Godot 语义增强打下非常扎实的基础。

---

## 16. 建议的下一步执行动作

建议实际开工时直接拆成 5 个工作包：

1. Runtime 状态机与调试快照
2. NetClient 重试与 fallback
3. ActionExecutor 回滚记录
4. Dock 撤销入口与提示流程
5. Context/Prompt 相关性排序

每个工作包都应单独有：

- 涉及文件
- 预期行为
- 人工验证点

如果执行顺利，v2.2 会是一个明显提升“稳定性和可信度”的版本，而不是单纯的功能版本。
