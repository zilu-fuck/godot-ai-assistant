# 必看

## 先说结论

这个仓库里真正要交付的核心是 [addons/ai_assistant](E:/test/addons/ai_assistant) 这个 Godot 编辑器插件。
`test/` 目录主要是测试和演示内容，不是这次 v2.0.1 版本的交付重点。

## 这是什么项目

这是一个嵌入 Godot 4 编辑器右侧 Dock 的 AI 编码助手插件。

它不是普通聊天窗口，而是一个能结合当前脚本、选区、场景、项目文件结构、Git 状态和规则文件工作的编辑器内 AI 助手。

## v2.0.1 当前已经具备的能力

- 多会话聊天，支持新建、切换、删除、清空
- API 配置和会话记录本地持久化
- 流式输出、手动停止、错误回显
- 自动读取当前脚本，或优先读取当前选区
- 自动注入场景路径、项目文件摘要、Git 摘要和规则内容
- 分层规则系统
- 支持内置规则、用户规则、项目规则、目录局部规则
- 支持 `@include`，并检测循环引用
- 结构化会话记忆与自动压缩
- Context Ring 上下文占用预估
- DeepSeek / OpenAI Compatible provider 抽象
- Apply 流程统一走 `ActionExecutor`
- 支持候选目标、diff 预览和高风险二次确认
- 支持从剪贴板读取报错并生成分析请求

## 当前架构怎么理解

可以把 v2.0.1 理解成 4 层：

1. UI 层
   [ai_dock.gd](E:/test/addons/ai_assistant/ai_dock.gd) 负责界面和交互。

2. Runtime 层
   [ai_runtime.gd](E:/test/addons/ai_assistant/core/ai_runtime.gd) 负责把“输入、历史、规则、上下文、provider、动作执行”串起来。

3. Prompt / Context / Memory 层
   负责构造模型真正收到的请求内容。

4. Provider / Action 层
   一边负责不同模型接口适配，一边负责把 AI 返回的代码安全地应用回编辑器。

## 模型切换与新增模型

当前内置模型定义在 [ai_provider_profiles.gd](E:/test/addons/ai_assistant/net/ai_provider_profiles.gd)：

- `deepseek-chat`
- `deepseek-reasoner`
- `openai-chat`

新增同类兼容模型，优先只改 profile。
如果是全新 payload 格式，再补 [ai_provider_adapter.gd](E:/test/addons/ai_assistant/net/ai_provider_adapter.gd)。

## 规则系统怎么用

当前会按优先顺序读取这些来源：

- 内置规则
- `user://ai_assistant_rules.md`
- `res://AI_ASSISTANT.md`
- 当前脚本目录向上的 `AI_ASSISTANT.local.md`

规则支持 `@include path`。
如果包含关系出现循环，会在加载错误里提示出来。

## Apply 流程现在是什么样

当前 AI 返回代码块后，不会直接写进编辑器，而是先通过 [ai_action_executor.gd](E:/test/addons/ai_assistant/actions/ai_action_executor.gd) 规划动作。

主要行为：

- 有选区时优先走替换选区
- 无选区时优先插入到光标
- 如果命中更合适的函数或文件目标，会先进入候选目标选择
- 真正执行前会先给 diff / review
- 高风险改动会要求二次确认

## 本地数据保存在哪里

- `user://ai_assistant_config.cfg`
- `user://ai_sessions.json`

这意味着：

- API Key 保存在本地用户目录，不会直接写回仓库
- 会话记录默认也不进 Git

## 当前边界

- 这仍然是编辑器内 AI 助手，不是游戏运行时 AI

- `ActionExecutor` 重点还是代码应用链路，不是通用 agent framework
