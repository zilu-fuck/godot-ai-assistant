# AI Assistant 最大化重构方案

## 文档状态

- 日期：2026-04-14
- 项目路径：`E:\test`
- 目标模块：`addons/ai_assistant`
- 目标：在尽可能保留现有插件外壳和可用功能的前提下，对当前 Godot AI 助手插件进行最大程度、最高质量的系统性重构

## 1. 执行摘要

当前插件已经具备基础可用性：它能在 Godot 编辑器内嵌聊天面板、支持多会话、支持流式输出、支持引用当前脚本上下文、支持代码块复制与应用、支持手动压缩对话、支持一键调试剪贴板报错。

但从架构层面看，它仍然属于“单控制器驱动”的实现方式。核心智能链路大多集中在 `ai_dock.gd` 中，导致以下问题：

- Prompt 构建方式比较原始，仍以字符串拼接为主。
- 运行时上下文没有分层管理，用户问题、代码上下文、系统规则混在一起。
- 长对话治理能力弱，当前只有人工触发的压缩，没有真正的结构化记忆体系。
- Provider 层抽象不充分，后续接入其他 API 或更细的能力差异会越来越难维护。
- 代码应用流程已有雏形，但还没有形成“动作协议层”。
- 当前实现已经开始触碰 Claude Code / Codex 这类 runtime 的核心问题，但内部结构还不足以支撑继续演进。

因此，如果目标是“最大程度重构”和“达到最好的效果”，最合适的策略不是在 `ai_dock.gd` 上继续修修补补，而是：

- 保留插件入口和现有可见交互外壳；
- 重建 assistant runtime；
- 将请求构建、上下文收集、规则加载、记忆压缩、动作执行、Provider 适配拆分成独立模块；
- 为后续的工具调用、上下文治理、规则体系和多 Provider 兼容打下基础。

这不是整个 Godot 项目的推倒重来。
这是一场针对插件核心智能运行时的“大重构”。

## 2. 当前状态评估

### 2.1 当前已经具备的能力

- 以 Godot `EditorPlugin` 形式挂载到编辑器右侧 Dock。
- 支持聊天会话的创建、切换、删除和持久化。
- 支持基于 HTTP 流式输出模型回复。
- 能自动读取当前脚本，或读取选中的代码片段作为上下文。
- 能渲染代码块，并提供 `Copy` 和 `Apply` 操作。
- 支持手动压缩历史对话，生成摘要。
- 支持读取剪贴板报错文本，自动拼装调试问题。

### 2.2 当前架构的主要问题

#### 问题 1：`ai_dock.gd` 责任过重

目前 `ai_dock.gd` 同时承担了：

- UI 事件响应
- Prompt 组装
- 上下文收集
- 会话状态修改
- 历史压缩流程
- 调试辅助流程
- Diff 预览与代码插入
- 调用网络模块发起请求

这意味着它既是视图控制器，又是 runtime，又是业务编排器，又是部分存储入口。这样的文件在项目继续扩展后会迅速成为维护瓶颈。

#### 问题 2：Prompt 构建没有层次

当前发送流程本质上是：

1. 读取输入框文本；
2. 读取当前脚本或选中代码；
3. 把代码上下文直接拼到最后一条用户消息后面；
4. 再根据模型模式决定是否加 system role；
5. 然后把最近若干条历史一起发出去。

这种做法在功能少的时候能跑，但一旦加入：

- 项目规则
- 用户个人规则
- Git 状态
- 项目地图
- 自动记忆摘要
- 动作协议
- 工具调用

就会迅速变得混乱，并造成调试困难。

#### 问题 3：缺少规则系统

当前插件只有一条较短的 Godot 哲学 system prompt，这远远不够。  
真正好用的 AI 编码助手，必须能持续读取并遵守：

- 默认内置规则
- 用户个人习惯
- 项目规范
- 局部目录规范

否则随着项目变大，模型会越来越不稳定。

#### 问题 4：长对话治理能力不足

当前虽然已有“压缩对话”按钮，这一点是很好的基础，但仍存在局限：

- 只能手动触发；
- 输出是普通文本摘要；
- 没有结构化记忆字段；
- 没有自动预算判断；
- 没有和后续上下文注入机制形成闭环。

#### 问题 5：Provider 层抽象偏薄

当前代码能跑 DeepSeek 兼容接口，但如果未来引入：

- OpenAI 风格接口
- Claude 风格消息格式
- 不同的 system role 支持策略
- reasoning 字段差异
- tool call 能力差异

就会导致主流程越来越多条件分支。

#### 问题 6：动作执行尚未成型

当前“Apply”已经是非常好的起点，但仍属于“从回复文本里抓代码块，然后插入编辑器”的阶段。  
更高阶的 runtime 应该能够识别模型输出的“动作意图”，例如：

- 替换当前选区
- 在光标处插入
- 重写当前函数
- 只展示 diff 不直接应用
- 只解释不修改

### 2.3 文档与实现存在不一致

README 中提到了：

- 项目树扫描
- 上下文去重
- 脏检查

但从当前代码实现看，这些能力还没有真正完整落地，或者没有形成明确的模块。  
这意味着本次重构不仅是代码升级，也包括“让实现与文档重新对齐”。

## 3. 重构目标定义

本次重构的目标不是单纯“让代码更整洁”，而是同时实现下面两个目标：

1. 架构升级：让插件内部从“单文件拼装器”升级为“可演进的 assistant runtime”。
2. 体验升级：让用户明显感知到回答更稳、记忆更强、上下文更准、应用代码更安全。

因此，目标应明确为：

- 将核心请求管线模块化；
- 引入分层 Prompt 构建；
- 建立项目/用户规则系统；
- 引入结构化记忆与自动压缩；
- 增加系统上下文注入能力；
- 建立 Provider 能力抽象；
- 建立动作执行层；
- 为未来工具调用与更复杂 agent 能力预留结构空间。

## 4. 重构范围等级判断

### 4.1 范围结论

本次重构应定义为：

- 对 Godot 主项目影响：小
- 对插件 UI 层影响：小到中
- 对插件内部架构影响：大
- 对插件核心行为链影响：大
- 对持久化数据结构影响：中

换句话说：

- 不是重写整个项目；
- 不是必须推翻整个 UI；
- 但几乎等同于重写插件的“智能大脑”。

### 4.2 哪些部分应保留

优先保留：

- `ai_assistant.gd` 的插件入口壳
- `ai_dock.tscn` 的主体布局
- `ai_chat_renderer.gd` 的基础渲染能力
- 当前用户可见的主要交互流程

### 4.3 哪些部分应重写或强拆

必须重点重构：

- `ai_dock.gd` 的核心请求管线
- `ai_storage.gd` 的持久化结构
- `ai_net_client.gd` 的 Provider 适配接口
- 历史压缩与上下文注入逻辑
- Apply / Diff 行为链

## 5. 目标架构

### 5.1 高层架构图

建议把当前插件演进为如下结构：

```text
EditorPlugin
  -> Dock Controller
    -> Runtime Facade
      -> Prompt Builder
      -> Rules Loader
      -> Context Builder
      -> Message Normalizer
      -> Memory Manager
      -> Provider Adapter
      -> Session Store
      -> Action Executor
      -> Diff Service
      -> Debug Service
```

### 5.2 目标职责划分

#### 1. Dock Controller

职责：

- 接收按钮点击、输入、下拉框切换等 UI 事件
- 更新聊天视图和状态
- 将用户意图转发给 runtime

不应再承担：

- 直接构造 API payload
- 直接做持久化
- 直接处理复杂上下文拼装

#### 2. Runtime Facade

职责：

- 统一编排发送、停止、压缩、调试、应用代码等流程
- 为 UI 提供一个稳定入口
- 隔离 UI 和底层服务实现

#### 3. Prompt Builder

职责：

- 将 system sections、system context、user runtime context、history 组装成最终请求结构
- 为不同 Provider 输出兼容格式

#### 4. Rules Loader

职责：

- 按层级加载规则文件
- 支持 `@include`
- 返回合并结果、来源、错误信息

#### 5. Context Builder

职责：

- 收集当前脚本、选区、当前场景、当前日期、最近记忆、项目地图、Git 状态等信息
- 构建“运行时上下文”

#### 6. Message Normalizer

职责：

- 清理空消息
- 合并连续同角色消息
- 限制超长上下文片段
- 防止重复插入 reminder

#### 7. Memory Manager

职责：

- 维护结构化记忆
- 决定何时自动压缩
- 将记忆摘要注入到后续上下文中

#### 8. Provider Adapter

职责：

- 将统一 runtime 请求映射到具体 Provider 接口格式
- 声明 Provider 能力差异
- 解析不同返回字段

#### 9. Session Store

职责：

- 管理配置、会话、版本迁移
- 统一所有持久化读写

#### 10. Action Executor

职责：

- 解析模型输出对应的动作
- 统一控制插入、替换、预览、解释等编辑行为

## 7. Prompt 与上下文重构方案

### 7.1 当前问题

当前 `_on_send_pressed()` 的逻辑本质上是直接拼接：

- 用户输入
- 最近历史
- 当前脚本或选中代码
- 一句 Godot system prompt

这种方式实现快，但很难扩展，也很难追踪到底是哪一层信息影响了模型输出。

### 7.2 目标 Prompt 模型

建议把请求组装为以下层级：

1. 静态系统规则
2. 动态系统 section
3. 系统上下文
4. 用户运行时提醒
5. 原始对话历史

一个理想化的抽象结构如下：

```ts
{
  system_sections: [
    "插件内置规则",
    "Godot 编码哲学",
    "当前模型策略规则",
    "项目规则摘要",
    "Git 状态等系统上下文"
  ],
  runtime_user_context: "<system-reminder>...</system-reminder>",
  history: [...recent history...],
  actions_or_tools: [...]
}
```

### 7.3 各层应放什么内容

#### 静态系统规则

放置相对稳定、不随当前会话变化的核心规则，例如：

- 你是 Godot 4 助手
- 优先使用场景驱动思想
- 不要鼓励硬编码路径
- 如果输出代码，应使用 `gdscript` 代码块
- 如果涉及修改，应尽量说明替换范围

#### 动态系统 section

放置依赖当前请求环境但仍属于系统侧的信息，例如：

- 当前模型是 Chat 还是 Reasoner
- 当前请求是普通问答、调试、压缩还是代码应用任务
- 当前是否启用了实验性功能

#### 系统上下文

放置运行环境信息，例如：

- 当前日期
- 当前打开的脚本路径
- 当前场景路径
- Git 当前分支和简要状态
- 项目地图摘要

#### 用户运行时提醒

放置与当前操作高度相关但又不属于用户自然语言问题本身的信息，例如：

- 当前选中的代码
- 当前脚本全文或摘要
- 当前会话结构化记忆
- 当前已加载规则来源
- 调试模式下的剪贴板报错摘要

#### 原始对话历史

只保留真正的问答历史和必要的结构化记忆痕迹，不要继续在这里塞大段环境拼接信息。

### 7.4 为什么一定要这样分层

因为这会直接改善：

- 指令优先级清晰度
- 长对话稳定性
- 运行时可调试性
- Provider 可迁移性
- 未来缓存与预算控制能力

## 8. 规则系统设计方案

### 8.1 为什么必须要有规则系统

如果插件只靠一条 system prompt 工作，那么随着项目规模变大，模型会越来越容易：

- 忘记项目约定
- 输出不符合架构风格的代码
- 在不同会话里风格漂移

因此，规则系统是“长期稳定效果”的关键。

### 8.2 推荐的规则层级

建议采用简化但实用的四层规则体系：

1. 插件内置默认规则
2. 用户全局规则
3. 项目级规则
4. 局部目录规则

推荐文件设计：

- 插件内置：由代码中维护，或内置在资源文件中
- 用户全局：`user://ai_assistant_rules.md`
- 项目规则：`res://AI_ASSISTANT.md`
- 局部规则：当前脚本附近目录下的 `AI_ASSISTANT.local.md`

### 8.3 `@include` 支持

建议支持简单的 include 机制，例如：

```md
@include res://docs/ai/ui_rules.md
@include user://ai_assistant/style_rules.md
```

需要注意：

- 要支持相对路径和 Godot 风格路径解析
- 要防止循环引用
- 要在出错时返回可调试的错误信息

### 8.4 Rules Loader 的输出结构

Rules Loader 不应只是返回一段字符串，而应返回结构化结果，例如：

- `merged_text`
- `sources`
- `load_errors`
- `include_graph`
- `priority_order`

这样在 UI 中也能展示“当前到底加载了哪些规则”。

### 8.5 第一版不做什么

第一版没必要做：

- 图形化的规则优先级拖拽面板
- 复杂模板语法
- 动态宏引擎

这些都可以后置。

## 9. 结构化记忆与长对话治理

### 9.1 当前压缩能力的优点

你现在已经有“手动压缩历史”的按钮，这非常好，因为说明项目已经意识到了上下文治理的重要性。

### 9.2 当前压缩能力的局限

目前压缩结果还是一段普通摘要文本，存在以下不足：

- 不利于后续系统化注入
- 不利于调试“模型到底记住了什么”
- 不利于做自动预算策略
- 不利于未来的高质量记忆演进

### 9.3 推荐的结构化记忆字段

建议每个 session 存储如下字段：

- `core_goals`：当前核心目标
- `decided_architecture`：已经决定的结构与方案
- `open_questions`：尚未解决的问题
- `bug_notes`：调试中发现的坑和结论
- `recent_files`：近期关键文件
- `last_compacted_at`：最后压缩时间

### 9.4 压缩策略

建议支持三种模式：

1. 手动压缩
2. 超过消息数阈值时自动压缩
3. 超过近似 token 预算时自动压缩

同时要保留：

- 最近若干轮原始对话
- 最近关键动作记录
- 必要的报错上下文

### 9.5 注入方式

不要仅仅把压缩结果当成一条普通 assistant 消息塞回历史。  
应该同时把结构化记忆作为独立的 runtime context 层注入后续请求。

### 9.6 效果收益

这样做的收益包括：

- 长会话稳定性显著提高
- 对话越长越不容易“变笨”
- token 使用更经济
- 更容易观察摘要质量

## 10. Provider 与网络层重构

### 10.1 当前网络层可复用的部分

`ai_net_client.gd` 里当前的：

- `HTTPClient` 连接方式
- 轮询 `poll()`
- SSE chunk 拆解
- `reasoning_content` 流式拼接

这些都可以继续复用，不需要全推翻。

### 10.2 需要升级的地方

应该增加 Provider Profile 和能力声明，例如：

- `supports_system_role`
- `supports_reasoning_delta`
- `supports_streaming`
- `supports_tool_calls`
- `supports_cache_hints`

### 10.3 Provider Adapter 的职责

建议新增 `ai_provider_adapter.gd`，负责：

- 接收统一的 runtime 请求结构
- 转换为具体 Provider payload
- 定义流式字段解析规则
- 定义错误映射规则

### 10.4 为什么现在就要做这层

哪怕你当前还是只用 DeepSeek，这层也值得做，因为它能避免把 Provider 差异继续污染主流程。

## 11. 动作执行层与代码应用重构

### 11.1 当前 Apply 流程评价

当前的 `Apply` 和 `Diff` 机制已经非常接近真正的“agent action”形态了，这是当前插件最值得继续发展的部分之一。

### 11.2 目标动作模型

建议将动作定义为结构化意图，而不是仅靠“从代码块中提取文本”。

推荐支持以下动作类型：

- `insert_at_caret`
- `replace_selection`
- `rewrite_current_function`
- `rewrite_current_file`
- `show_diff_only`
- `explain_only`

### 11.3 渐进式改造方案

#### 第一阶段

- 保留当前代码块 `Apply`
- 但在内部统一走 `ActionExecutor`

#### 第二阶段

- 支持模型按约定标签或 JSON 返回动作元信息
- 解析出动作后再决定如何应用

#### 第三阶段

- 根据 Provider 能力，考虑演进到 tool-like action schema

### 11.4 安全要求

动作执行层必须遵守：

- 替换选区前必须允许预览
- 大范围替换必须默认走 diff 审查
- 只解释不修改的回答不应误触发应用行为
- 每次动作都应记录到会话历史或动作日志中

## 12. 项目上下文与项目索引

### 12.1 为什么值得做

真正好用的 Godot 助手，不应该只看当前脚本。
它应该至少能知道：

- 项目里有哪些核心脚本
- 哪些场景和脚本相关
- 近期这个会话中讨论过哪些文件

### 12.2 项目索引器建议

增加 `ai_project_indexer.gd`，初步扫描：

- `.gd`
- `.tscn`
- `.gdshader`

默认排除：

- `.godot`
- 导入缓存目录
- 无关缓存资源
- 视情况排除第三方 `addons`

### 12.3 第一版能力边界

第一版不需要做完整 AST 或符号分析。  
只要能实现：

- 文件存在性索引
- 基础项目地图摘要
- 近期相关文件聚合

就已经足够产生明显收益。

### 12.4 注入原则

项目地图不能每次把完整树结构都塞给模型。  
应该只传摘要，或按当前文件和当前会话筛选相关部分。

## 13. 持久化与数据迁移方案

### 13.1 为什么必须考虑迁移

当前持久化内容比较简单：

- `user://ai_assistant_config.cfg`
- `user://ai_sessions.json`

但重构后，会话对象很可能新增：

- schema 版本号
- 结构化记忆
- 动作历史
- 规则来源快照
- 上下文元数据

如果不做迁移设计，老用户的会话会有较大风险。

### 13.2 迁移策略

建议引入：

- `schema_version`
- `migration runner`
- 初次迁移备份

迁移流程建议如下：

1. 读取旧格式数据
2. 检测 schema 版本
3. 在内存中转换为新结构
4. 写入新 schema
5. 在第一次升级时保存旧文件备份

### 13.3 兼容性目标

必须保证：

- 旧配置仍可读取
- 旧聊天记录不丢失
- 标题和历史消息尽量保持原样
- 迁移失败时也不直接覆盖原数据

## 14. UI / UX 调整建议

### 14.1 应该新增的 UI 能力

建议在后续 UI 中增加：

- 当前生效规则来源的可见提示
- 当前会话是否已压缩、何时压缩过
- 当前 Provider / Model 能力摘要
- 当前上下文来源预览
- 更清晰的 diff 弹窗标题与动作说明

### 14.2 可以后做的 UI 能力

可以延后：

- Prompt Inspector 完整面板
- token 预算图表
- 规则优先级图形拖拽

### 14.3 核心原则

不要让 UI 改造反过来阻塞 runtime 重构。  
优先把内部大脑清干净，再逐步把可见信息暴露给用户。

## 15. 测试与验证方案

### 15.1 为什么需要更严格的验证

这次重构已经不再是单纯的样式修改或局部功能优化，而是核心请求链路重构。  
如果没有系统化验证，非常容易出现：

- 功能没坏，但回答质量下降
- 会话没丢，但迁移后行为异常
- UI 正常，但上下文层次错了

### 15.2 最小验证矩阵

#### 核心发送流程

- 普通文本提问
- 带选区提问
- 带当前脚本全文提问
- API Key 为空时阻止发送

#### 流式输出

- 正常结束
- 手动停止
- 非法 chunk 容错
- 网络断开处理

#### 会话管理

- 新建会话
- 删除会话
- 切换会话
- 重启后恢复会话

#### 压缩与记忆

- 短对话不压缩
- 手动压缩成功
- 压缩失败提示
- 结构化记忆重启后仍保留

#### 规则系统

- 用户规则加载
- 项目规则加载
- 本地规则加载
- `@include` 解析
- include 循环保护

#### 动作系统

- 代码块 Apply
- 选区替换前 diff 审查
- 光标插入
- 动作日志写入

#### Debug 流程

- 剪贴板为空
- 剪贴板报错 + 当前代码上下文
- 调试请求正确注入上下文

#### 数据迁移

- 读取旧版 session
- 迁移到新版 schema
- 标题与历史保留

### 15.3 推荐增加的调试工具

建议在开发阶段提供内部调试能力，例如：

- 最终 payload 预览
- 规则加载结果预览
- 当前记忆状态预览
- Fake Stream 回放工具

这样在重构过程中，能更快判断是 Prompt 问题、上下文问题，还是网络解析问题。

## 18. 主要风险与缓解策略

### 风险 1：重构失控，变成无边界重写

缓解方式：

- 严格按阶段推进
- 每阶段都保留可运行状态
- 每阶段都有退出标准

### 风险 2：会话迁移损坏历史数据

缓解方式：

- 引入 schema version
- 第一次迁移前备份
- 对旧数据准备专门样本做验证

### 风险 3：架构更复杂了，但回答效果没有变好

缓解方式：

- 提供 payload 预览工具
- 拿当前版本做质量对照
- 每个阶段评估“结构收益”和“实际体验收益”

### 风险 4：项目索引导致上下文膨胀

缓解方式：

- 只注入摘要
- 只注入相关切片
- 引入预算控制

### 风险 5：Provider 抽象过早过度设计

缓解方式：

- 只抽象现实已经存在的差异点
- 不提前为未知能力做复杂框架

## 19. 成功标准

只有当以下条件都满足时，本次重构才算真正成功：

- `ai_dock.gd` 不再是插件的“智能大脑”
- Prompt 构建具备明确分层
- 规则系统可加载、可追踪、可调试
- 长对话通过结构化记忆显著变稳
- 动作执行比现在更安全、更明确
- Provider 差异被隔离在专门模块
- Session 存储具备版本与迁移能力
- 文档与实现一致
- 后续新增智能能力时，不再需要继续堆满一个巨型控制器文件

## 20. 不应该优先做的事情

本次重构开始时，不应把精力优先放在以下内容上：

- 整个 Dock 界面的重新美术设计
- 多 agent / 协调器系统
- 厂商专用 prompt cache 技巧
- 完整 AST 级别项目分析
- 过度复杂的工具生态

这些方向以后都可以做，但不应该喧宾夺主，抢走对 runtime 重构最关键的资源。

## 21. 推荐的第一步落地动作

如果现在就开始动手，最值得做的第一步是：

1. 新建 `ai_runtime.gd`
2. 新建 `ai_prompt_builder.gd`
3. 把 `_on_send_pressed()` 里的请求构建流程搬出去
4. 暂时保持当前 UI 和当前 Provider 不变

为什么是这一步：

- 架构杠杆最大
- 破坏面相对可控
- 能最快把“单控制器逻辑”打散
- 为后续规则、记忆、Provider、动作系统都预留了接口

## 22. 最终建议

这个项目现在非常适合做一次“最大化但有节制”的重构。

它的优点是：

- 体量还不大
- 功能边界相对集中
- 主要问题都集中在插件内部
- 当前功能已经足够作为基线

这意味着现在动手，收益很高，风险也还可控。

最正确的思路不是：

- 继续往 `ai_dock.gd` 塞逻辑
- 再写几句更长的 system prompt
- 用局部 patch 的方式硬撑未来需求

而应该是：

- 保留外壳
- 重写核心运行时
- 小心迁移数据
- 分阶段推进
- 每一阶段都同时改善结构质量和用户感知效果

如果这套方案执行得好，这个插件会从“Godot 里能聊天的 AI 面板”，进化成“真正具备 Godot 原生协作能力的编辑器级 AI 助手”。
