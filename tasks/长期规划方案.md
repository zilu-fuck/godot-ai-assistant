# Godot AI Assistant 长期规划方案

## 1. 文档目的

本文用于给 `addons/ai_assistant` 提供一份长期可执行的演进路线。

这份规划不是照搬 Claude/Codex 一类通用 Agent CLI 的完整形态，而是吸收其中已经被验证有效的部分：

- 运行时分层
- 上下文工程
- 受控动作执行
- 状态与恢复
- 安全边界
- 可观测性与回归验证

然后结合 Godot 编辑器插件的真实使用场景，形成一条更适合本项目的路线。

结论先行：

- 这个项目应继续坚持“编辑器内受控 AI 助手”定位
- 不应过早演化成“什么都能做的通用 Agent 框架”
- 最核心竞争力应放在：
  - 更懂 Godot 项目与编辑器上下文
  - 更安全地把建议落回脚本和场景
  - 更稳定地维护长会话、规则和历史
  - 更容易被自己长期维护，而不是堆功能

---

## 2. 产品长期定位

### 2.1 核心定位

本项目的长期定位应是：

一个深度嵌入 Godot 编辑器工作流的 AI 开发助手，而不是一个放在 Dock 里的聊天窗口。

它应该擅长的事情是：

- 理解当前脚本、当前选区、当前场景、项目结构、Git 状态、规则和会话历史
- 输出对 Godot 开发真正有帮助的解释、修改建议和结构化操作
- 以安全、可审查、可回滚的方式把 AI 建议应用回工程

### 2.2 不应偏离的边界

长期演进过程中，应明确避免以下方向失控：

- 不把插件做成任意 shell / 任意系统操作入口
- 不急于复制完整 MCP / 多 Agent / 云端任务系统
- 不把“Apply 代码”直接升级成“无审查自动执行”
- 不为了支持更多模型而破坏运行时抽象
- 不让 UI 层重新变成业务和策略堆积点

### 2.3 长期价值主张

如果项目持续做深，最重要的差异化价值应来自这几项：

1. 对 Godot 编辑器语境的理解优于通用聊天产品
2. 对 GDScript / `.tscn` / 场景树 / 节点关系更友好
3. 对“建议如何安全应用”有强控制力
4. 对长期维护者足够透明、可调试、可测试

---

## 3. 长期设计原则

后续所有改动都建议遵守下面这些原则。

### 3.1 受控优先于炫技

宁可能力少一点，也要保证：

- 可解释
- 可审查
- 可取消
- 可恢复
- 可回滚

### 3.2 优先建设运行时，不优先堆 UI

真正决定项目质量的不是按钮数量，而是：

- 请求如何构造
- 上下文如何选择
- 风险如何判断
- 修改如何应用
- 出错如何恢复

### 3.3 上下文质量优先于模型数量

短期内，一个上下文构建更合理、动作链路更安全的插件，通常会比“支持更多模型名字”的插件更有价值。

### 3.4 分层清晰优先于功能集中

必须持续保持下面的职责边界：

- UI 层负责交互与展示
- Runtime 负责编排与状态
- Prompt / Context / Rules / Memory 负责输入构造
- Provider 层负责模型适配
- ActionExecutor 负责安全落地

### 3.5 任何新能力都必须回答 3 个问题

1. 它如何被测试？
2. 它如何失败并恢复？
3. 它会不会削弱现有安全边界？

---

## 4. 当前架构评价

从 v2.0.1 状态看，当前已经具备比较好的基础：

- `ai_dock.gd` 主要负责 UI 和交互
- `core/ai_runtime.gd` 已承担主链路编排
- `prompt/`、`memory/`、`project/`、`net/`、`actions/` 已形成模块边界
- `ActionExecutor` 已成为 Apply 流程的统一入口
- 已具备规则系统、结构化记忆、上下文预算、diff 预览和高风险确认

这说明项目已经完成“从脚本堆逻辑到有运行时骨架”的关键阶段。

但从长期维护角度看，还存在几个下一阶段必须补强的地方：

1. Runtime 仍偏流程式，状态机语义还不够强
2. 上下文选择还偏静态预算，相关性建模可以更强
3. 记忆系统仍偏摘要桶，尚未形成更稳定的结构语义
4. Apply 流程缺少更完整的回滚与变更追踪
5. 网络层缺少更完整的重试 / 降级 / 部分响应恢复
6. 测试和可观测性还没有成为与功能同等级的投入项

## 5. 长期目标图谱

建议把长期目标分成六条主线并行推进。

### 5.1 主线 A：Runtime 产品化

目标：

- 把 `AIRuntime` 从“请求编排器”升级为“稳定运行时内核”

长期应达到的状态：

- 明确状态机
- 明确事件流
- 明确失败恢复
- 明确数据输出结构
- 不把策略判断留给 Dock 临时推断

### 5.2 主线 B：上下文与记忆增强

目标：

- 让模型收到的内容更少但更准

长期应达到的状态：

- 上下文能按相关性动态排序
- 记忆具有明确类别和可信度
- 当前脚本、当前场景、错误信息、项目结构之间能协同工作

### 5.3 主线 C：安全 Apply 与可回滚修改

目标：

- 让“应用 AI 建议”成为可依赖的编辑器能力，而不是高风险捷径

长期应达到的状态：

- 所有修改都有计划、预览、确认、执行、记录、回滚
- 风险评估透明
- 高风险操作永远需要显式确认

### 5.4 主线 D：Godot 语义增强

目标：

- 让插件越来越像“懂 Godot”的助手，而不是“会写代码的聊天框”

长期应达到的状态：

- 理解场景树
- 理解节点与脚本挂载关系
- 理解 `.tscn` 与 `.gd` 的联动
- 理解项目内资源路径与常见错误模式

### 5.5 主线 E：可观测性、测试与维护体系

目标：

- 让项目可以长期迭代，而不是只能靠作者记忆维持

长期应达到的状态：

- 有足够日志
- 有稳定回归样例
- 有 mock provider
- 有 schema 迁移策略
- 有升级清单和发布流程

### 5.6 主线 F：谨慎引入受限工具化

目标：

- 只在 Godot 编辑器内引入高价值、强约束的工具能力

长期应达到的状态：

- 先只读，再受控写
- 优先编辑器内能力，不扩成系统级能力
- 所有工具调用都在项目边界内可解释

## 6. 三阶段路线图

## 阶段一：v2.2 稳态增强期

### 阶段目标

把当前架构从“已经分层”推进到“稳定可维护”。

### 重点事项

#### 6.1 Runtime 状态机化

将 `AIRuntime` 显式拆成阶段：

- `idle`
- `collect_context`
- `load_rules`
- `build_request`
- `send_request`
- `streaming`
- `normalize_response`
- `plan_action`
- `await_review`
- `applying`
- `persisting`
- `completed`
- `failed`
- `stopped`

收益：

- 更容易排查问题
- 更容易向 UI 输出结构化状态
- 更容易写测试和恢复逻辑

#### 6.2 请求与流式调用增强

增强 `AINetClient` 与 Runtime 协作：

- 增加可选单次重试
- 流式失败时允许降级为非流式请求
- 保留已接收内容，支持“部分响应提示”
- 更明确地区分连接失败、首包超时、流中断、服务端异常

#### 6.3 Apply 可回滚

对 `ActionExecutor` 增加最小回滚体系：

- 记录目标路径
- 记录变更前文本
- 记录变更后文本
- 记录动作类型和风险级别
- 提供“一键撤销上次 AI 改动”的能力入口

#### 6.4 上下文相关性排序第一版

在现有 budget 基础上增加“相关性评分”：

- 用户 prompt 关键词
- 当前活动文件
- 最近编辑文件
- 最近失败文件
- 当前场景路径

不再只按固定优先级选择上下文。

#### 6.5 调试与可观测性第一版

在调试视图中增加：

- 当前 runtime 状态
- 实际入选的上下文项
- 被裁剪项与原因
- provider 能力声明
- 当前动作计划摘要
- 最近一次失败原因

### 阶段验收标准

- 运行时状态更可读
- Apply 失败后不至于让用户丢失上下文
- 调试信息可以解释大部分“为什么这样做”
- 回归清单从“经验”变成“稳定流程”

---

## 阶段二：v2.3 Godot 语义增强期

### 阶段目标

让插件在 Godot 项目中的理解能力显著提升。

### 重点事项

#### 6.6 场景树上下文

上下文系统加入场景结构信息：

- 根节点
- 关键子节点
- 节点类型
- 挂载脚本
- 关键资源路径

这会对 UI 脚本、节点逻辑、场景搭建类请求非常有帮助。

#### 6.7 符号级项目索引

升级 `AIProjectIndexer`：

- 记录类名、函数名、信号、导出变量
- 记录 `.tscn` 对 `.gd` 的挂载关系
- 记录常见资源类型
- 支持按符号或路径匹配上下文

目标不是做完整语言服务器，而是做“足够便宜、足够有用”的语义索引。

#### 6.8 错误驱动工作流

建设“从报错到修复”的直通路径：

- 读取剪贴板错误
- 读取当前编辑器报错
- 定位可能文件
- 自动拼装修复请求
- 允许一键进入修复模式

#### 6.9 多工作模式

不要只依赖自由 Prompt，增加稳定入口：

- 解释当前脚本
- 修复当前报错
- 重构当前函数
- 为选中节点生成脚本
- 创建场景骨架
- 为当前脚本补注释或类型

#### 6.10 记忆系统升级

把记忆拆成更稳定的类别：

- 当前任务目标
- 已采纳决策
- 未解决问题
- 最近失败记录
- 最近相关文件
- 项目长期偏好

### 阶段验收标准

- 模型对 Godot 项目结构更“有感知”
- 同样的 prompt，建议更贴合当前项目和场景
- 错误修复和局部改动的命中率明显提高

---

## 阶段三：v3.0 受限 Agent 化期

### 阶段目标

在不突破安全边界的前提下，把插件从“上下文增强聊天助手”升级成“受限编辑器 Agent”。

### 核心原则

- 只做编辑器内高价值工具
- 只做强约束工具
- 先只读，后受控写
- 不引入系统级自由执行能力

### 重点事项

#### 6.11 内部受限工具集

优先考虑的内部工具：

- 读取当前脚本
- 读取当前选区
- 读取当前场景树
- 查找项目中的脚本/场景/资源
- 打开资源并定位
- 创建脚本草稿
- 创建场景草稿

其中前四项适合优先做成只读能力。

#### 6.12 工具调用协议最小化

如果未来允许模型显式发起工具调用，必须满足：

- 工具 schema 明确
- 输入参数可验证
- 输出结构固定
- 调用记录可查看
- 受统一权限策略控制

#### 6.13 多步工作流但不引入复杂多 Agent

可以做的工作流：

- 先收集上下文，再给方案，再等待用户确认，再应用
- 先扫描目标文件，再选择候选目标，再生成 patch

不建议此阶段就引入：

- 后台多代理并行
- 自动级联修改多个文件且无明确审查
- 会话外长期自治任务

### 阶段验收标准

- 插件可以完成更复杂但仍受控的编辑器工作流
- 新增工具能力不会让安全性和可维护性失控

## 7. 关键模块长期演进建议

## 7.1 `ai_runtime.gd`

长期目标：

- 成为整个插件的唯一运行时真相源

建议演进：

- 显式状态机
- 统一事件输出
- 请求生命周期记录
- 动作规划结果标准化
- 错误分类标准化
- 会话写入时机统一

不建议：

- 让 Dock 再去解释 Runtime 的意图
- 把太多 UI 条件判断塞回 Runtime 之外

## 7.2 `ai_prompt_builder.gd`

长期目标：

- 成为“为什么模型看到这些内容”的唯一解释层

建议演进：

- 相关性评分
- per-kind 预算与全局预算协同
- 历史消息预算分层
- 更稳定的上下文裁剪说明
- 针对不同 provider 能力切换组装策略

## 7.3 `ai_context_builder.gd`

长期目标：

- 从“上下文收集器”升级为“编辑器环境理解器”

建议演进：

- 增加场景树与节点信息
- 增加当前错误、最近编辑、最近失败文件
- 增加符号级摘要
- 增加上下文来源可信度与更新时间

## 7.4 `ai_memory_manager.gd`

长期目标：

- 从“压缩历史”升级为“结构化长期协作记忆”

建议演进：

- 细分类别
- 明确来源
- 明确过期策略
- 明确 schema 迁移
- 记录用户采纳/拒绝历史

## 7.5 `ai_action_executor.gd`

长期目标：

- 成为编辑器内 AI 修改的统一安全执行面

建议演进：

- patch 优先
- 标准化 review 数据
- 回滚记录
- 高风险因子明细
- 分步应用
- 场景和脚本操作统一落在动作计划内

## 7.6 `ai_provider_profiles.gd` / `ai_provider_adapter.gd`

长期目标：

- 支撑多 provider，而不污染上层运行时

建议演进：

- 能力声明更完整
- payload 构建更明确
- 对 JSON mode / response format 的能力预留
- 对 reasoning / 非 reasoning 模型分流

## 7.7 `ai_net_client.gd`

长期目标：

- 成为“可恢复、可诊断、可兼容”的网络层

建议演进：

- 单次重试
- 非流式 fallback
- 错误分类码
- SSE 解析异常保护
- 调试级请求摘要

## 8. 安全与权限长期策略

虽然这是本地插件，但长期仍然必须把安全当成一等公民。

### 8.1 长期必须坚持的底线

- 默认不做静默写入
- 默认不做无预览跨文件修改
- 默认不信任高风险整文件替换
- 默认不暴露系统级执行能力

### 8.2 风险分级建议

低风险：

- 光标处插入
- 当前选区替换
- 仅解释、不写入

中风险：

- 当前脚本函数级替换
- 新建脚本
- 当前场景下新建资源

高风险：

- 整文件替换
- 跨文件写入
- 覆盖 `.tscn`
- 修改配置文件
- 删除大段代码

### 8.3 后续可选增强

- 受保护路径白名单/黑名单
- 脱敏日志
- 会话导出过滤 API Key / token / URL 参数
- 更清楚的风险说明文本

---

## 9. 测试与验证长期规划

如果没有测试体系，这个项目越强越难维护。

### 9.1 验证分层

建议长期维持四层验证：

1. 静态检查
2. Headless 编辑器加载验证
3. Mock Provider 行为回归
4. 关键模块的固定输入输出测试

### 9.2 最值得固定样例的模块

- Rules Loader
- Prompt Builder
- Memory Manager
- Action Executor
- Provider Adapter
- Net Client

### 9.3 Mock 响应样本建议

建议维护一组固定 SSE/JSON 样本：

- 普通解释回复
- 带可应用代码块的回复
- 大段整文件高风险回复
- 场景创建回复
- 401 / 403 / 500 错误
- 空响应
- 流中断
- reasoning delta 与 content delta 混合

### 9.4 回归清单制度化

将现有 `runtime_adaptation_checklist.md` 持续扩充并纳入发布流程。

建议把以下改动强制关联回归清单：

- Runtime 改动
- ActionExecutor 改动
- Prompt / Context 改动
- Provider / NetClient 改动
- Scene Apply 改动

---

## 10. 可观测性与调试长期规划

长期来看，一个 AI 插件如果不能解释自己，维护成本会迅速上升。

建议逐步补齐以下信息：

- 当前 runtime 阶段
- 最近一次请求 ID
- provider / model / capabilities
- 实际使用的上下文项
- 被丢弃的上下文项
- 动作计划
- 风险判定依据
- 执行结果
- 最近一次失败原因

调试信息不一定全部暴露给普通用户，但必须对维护者可用。

---

## 11. 文档与维护机制

为了长期维护，建议固定维护这几类文档。

### 11.1 面向用户

- README
- 功能说明
- 使用方式
- 风险说明

### 11.2 面向维护者

- 架构说明
- 模块边界
- 数据结构与 schema
- 回归清单
- 版本迁移说明

### 11.3 面向未来自己

每个版本发布时，至少记录：

- 本次变更了什么
- 为什么这样改
- 改动影响哪些模块
- 哪些边界仍未突破
- 下一步最合理的续点是什么

---

## 12. 版本节奏建议

建议采用“能力建设版本”和“体验优化版本”交替节奏。

### 12.1 能力建设版本

特点：

- 改 Runtime
- 改 Context
- 改 Memory
- 改 ActionExecutor
- 改 Provider/Net 层

要求：

- 必须强化测试
- 必须强化验证
- 必须更新回归清单

### 12.2 体验优化版本

特点：

- 改 Dock
- 改工作模式入口
- 改可视化和交互
- 改提示文案和流程

要求：

- 不破坏核心运行时边界
- 不在 UI 层偷塞策略逻辑

---

## 13. 资源投入优先级

如果时间有限，长期优先级建议如下。

### P0：必须优先

- Runtime 状态机化
- Apply 可回滚
- 调试信息增强
- 上下文相关性排序
- 网络层重试与降级
- 会话 schema 迁移准备

### P1：应尽快做

- 场景树上下文
- 符号级项目索引
- 错误驱动工作流
- 多工作模式入口
- 更稳定的结构化记忆

### P2：可以规划但不抢先

- 受限工具调用协议
- 更复杂的半自动工作流
- 更多 provider 特性协商
- 插件内更丰富的操作编排

### P3：谨慎观望

- 多 Agent
- 后台自治任务
- 通用插件生态
- 系统级能力扩展

## 14. 最终判断

这个项目最好的未来，不是成为一个缩小版 Claude CLI，也不是成为一个普通聊天插件。

它更适合成为：

一个在 Godot 编辑器内稳定、受控、可解释、可回滚、真正懂项目上下文的 AI 开发助手。

因此，长期路线应始终围绕下面四句话展开：

1. 让 Runtime 更稳定
2. 让上下文更准确
3. 让 Apply 更安全
4. 让维护更轻松

如果这四点持续做深，项目会越来越像一个成熟产品。
如果偏离这四点而去追逐“万能 Agent”形态，复杂度大概率会先于价值增长。
